# 项目部署说明

## 1. 先说结论

这个项目是一个 **Hexo + NexT** 的静态博客项目，当前已经配置成：

- 源码仓库：当前这个仓库
- 构建命令：`npm run build`
- 发布命令：`npm run deploy`
- 发布目标：GitHub Pages
- 发布分支：远端仓库 `Captain-Bridge/Captain-Bridge.github.io` 的 `master`
- 自定义域名：`baixiao.club`

也就是说，你现在最适合继续沿用的部署方式是：

1. 在本地修改 `source`、配置文件、主题样式
2. 本地执行 Hexo 构建
3. 执行 `hexo deploy`，把生成后的静态文件推到 GitHub Pages 仓库
4. 由 GitHub Pages 对外提供网站访问

## 3. 当前部署方式的工作原理

这套项目不是“构建后直接跑 Node 服务”，而是：

1. Hexo 读取 `source/`、主题配置和脚本
2. 生成纯静态文件到 `public/`
3. `hexo-deployer-git` 把 `public/` 的内容推到目标 GitHub 仓库
4. GitHub Pages 直接托管这些 HTML/CSS/JS/图片文件

所以线上不需要：

- PM2
- Node 常驻进程
- 数据库服务
- 后端 API

从部署角度看，它本质上就是一个静态站点。

## 6. 新机器第一次部署时要注意什么

### 6.1 GitHub 权限

当前部署仓库走的是 HTTPS：

- `https://github.com/Captain-Bridge/Captain-Bridge.github.io.git`

所以在新机器上第一次 `npm run deploy` 时，你需要保证当前 Git 有权限推送这个仓库。

常见做法有两种：

#### 做法 1：继续用 HTTPS

- 让 Git 登录有该仓库权限的 GitHub 账号
- 使用 Git Credential Manager 或 Personal Access Token

#### 做法 2：改成 SSH

把 [`_config.yml`](C:\codes\myblog\_config.yml) 里的 `deploy.repo` 改成 SSH 地址，例如：

```yml
deploy:
  type: git
  repo:
    github: git@github.com:Captain-Bridge/Captain-Bridge.github.io.git
  branch: master
```

然后确保本机 SSH key 已加到 GitHub。

如果你长期在自己的电脑上维护，我更建议 SSH，省去反复验证账号。

### 6.2 不要手改 `public/` 和 `.deploy_git/`

这两个目录都是部署产物或部署缓存：

- `public/`：构建结果
- `.deploy_git/`：发布缓存仓库

日常不要把它们当源码改。真正应该维护的是：

- `source/`
- `_config.yml`
- `_config.next.yml`
- `themes/` 下你自己改过的内容

## 7. 和这个项目强相关的部署注意点

### 7.1 这个站点默认按“根域名站点”部署

我检查到首页和主题里有不少绝对路径，例如：

- `/images/...`
- `/marathon-lore/...`

再加上 [`scripts/force-root-home.js`](C:\codes\myblog\scripts\force-root-home.js) 会强制生成根路径首页，所以这套站点更适合部署在：

- 顶级域名，如 `https://baixiao.club`
- 或 GitHub 用户主页域名，如 `https://用户名.github.io`

不太适合直接部署成：

- `https://域名/某个子目录/`
- `https://用户名.github.io/仓库名/`

否则这些以 `/` 开头的资源路径很容易失效。

如果你未来想改成“子路径部署”，需要一起调整：

- [`_config.yml`](C:\codes\myblog\_config.yml) 里的 `url`
- 首页里的静态资源引用
- 主题配置中的资源路径
- 可能还要检查 `source/about/`、`source/marathon-lore/` 下手写页面

### 7.2 自定义域名已经写进构建产物

[`source/CNAME`](C:\codes\myblog\source\CNAME) 当前内容是：

```text
baixiao.club
```

这意味着每次构建发布时，生成站点里都会带上这个 `CNAME` 文件。

如果你未来更换域名，记得至少同时改两处：

1. [`source/CNAME`](C:\codes\myblog\source\CNAME)
2. [`_config.yml`](C:\codes\myblog\_config.yml) 里的 `url`

### 7.3 首页是手写页面，不是默认 Hexo 列表页

[`source/index.html`](C:\codes\myblog\source\index.html) 是手写首页，且 [`_config.yml`](C:\codes\myblog\_config.yml) 里把默认首页生成器路径改成了：

```yml
index_generator:
  path: marathon-index-disabled
```

这说明：

- 根首页不是普通博客列表
- 你不能只按“标准 Hexo 博客”思路理解这个项目

部署本身没问题，但如果以后你发现“Hexo 文章没自动出现在首页”，这是当前设计，不是部署失败。

### 7.4 部分页面可能是直接跳过渲染的

[`_config.yml`](C:\codes\myblog\_config.yml) 中有：

```yml
skip_render:
  - index.html
  - about/index.html
  - marathon-lore/content/**
```

这表示部分文件会原样进入产物目录，因此：

- 如果这些文件里用了绝对路径，部署路径必须匹配
- 如果这些文件编码有问题，线上也会原样带出去

## 9. 如果你不想走 GitHub Pages，还有哪些部署方式

因为这个项目最终产物就是 `public/` 里的静态文件，所以也可以部署到任何静态托管平台。

### 方案 B：部署到 Nginx / 宝塔 / 云服务器

流程：

1. 本地执行 `npm run build`
2. 把 `public/` 全部上传到服务器站点目录
3. 用 Nginx 把站点根目录指向这些文件

Nginx 核心思路就是把：

- 网站根目录指向构建后的静态目录
- 首页默认文件设为 `index.html`

这种方式适合：

- 你自己有云服务器
- 你希望完全掌控域名、HTTPS、缓存策略

### 方案 C：部署到 Netlify / Vercel / Cloudflare Pages

也可以直接接 Git 仓库自动构建。

典型配置思路：

- Build command: `npm run build`
- Publish directory: `public`

但这套项目前提是你要注意“根路径部署”问题，因为你现在大量资源写的是绝对路径。

如果平台把站点发布在子路径下，页面可能会错。
